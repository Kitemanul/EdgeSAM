# EdgeSAM 模型扩充方案

## 背景

在运动员数据集上微调 EdgeSAM 后，大目标性能提升 5 个点，但小目标性能下降 20 个点。需要在控制参数量（端侧部署）的前提下扩充模型能力，重点改善小目标表现。

## 问题分析

小目标掉点严重的核心原因：**高分辨率空间信息丢失**。

当前 RepViT m1 编码器经过 4 次 2x 下采样（总共 16x），最终 feature map 是 64x64。fuse 机制只融合了 stage2 (192ch, 64x64) 和 stage3 (384ch, 64x64)，**丢失了 stage0 (48ch, 256x256) 和 stage1 (96ch, 128x128) 的高分辨率细节**。

对于小目标来说，一个 32x32 像素的目标在 64x64 的 feature map 上只有 2x2，信息非常稀疏。而 stage0/stage1 中保留了丰富的边界和纹理细节。

### 当前 RepViT m1 数据流

```
1024x1024x3
  → Stem (4x↓)         → 256x256x48
  → Stage 0 (3 blocks)  → 256x256x48    ← 高分辨率，未被利用
  → Stage 1 (4 blocks)  → 128x128x96    ← 中分辨率，未被利用
  → Stage 2 (16 blocks) → 64x64x192     ← 参与 fuse
  → Stage 3 (3 blocks)  → 64x64x384     ← 参与 fuse（skip_downsample）
  → Fuse                → 64x64x256     (stage2 + stage3)
  → Neck                → 64x64x256     (最终 image embedding)
```

### 当前 Fuse 参数量

| 模块 | 结构 | 参数量 |
|---|---|---|
| fuse_stage2 | Conv2d(192, 256, 1x1, bias=False) | 49,152 |
| fuse_stage3 | Conv2d(384, 256, 1x1, bias=False) + Upsample(2x) | 98,304 |
| **合计** | | **147,456 (~147K)** |

## 方案一：多尺度特征融合（推荐优先做）

### 核心思路

将 stage0 和 stage1 的高分辨率特征融合到最终输出中，补回小目标的空间细节信息。

### 子方案对比

#### 方案 1A：1x1 投影 + AvgPool 下采样（推荐，最轻量）

```
fuse_stage0: Conv2d(48, 256, 1x1, bias=False) + AvgPool2d(kernel=4, stride=4)
             参数: 48 × 256 = 12,288

fuse_stage1: Conv2d(96, 256, 1x1, bias=False) + AvgPool2d(kernel=2, stride=2)
             参数: 96 × 256 = 24,576
```

| 指标 | 值 |
|---|---|
| 新增参数 | **36,864 (~37K)** |
| 占模型总参数比 | 0.38% |
| 端侧友好度 | 非常友好，AvgPool 所有框架原生支持 |
| ONNX/CoreML 兼容 | 无额外算子压力 |

#### 方案 1B：1x1 投影 + Depthwise Conv 下采样（可学习下采样）

```
fuse_stage0: Conv2d(48, 256, 1x1) + DWConv(256, 3x3, s=2) + DWConv(256, 3x3, s=2)
             参数: 12,288 + 2,304 + 2,304 = 16,896

fuse_stage1: Conv2d(96, 256, 1x1) + DWConv(256, 3x3, s=2)
             参数: 24,576 + 2,304 = 26,880
```

| 指标 | 值 |
|---|---|
| 新增参数 | **43,776 (~44K)** |
| 占模型总参数比 | 0.46% |
| 端侧友好度 | 友好，DWConv 端侧高效 |
| 优势 | 下采样可学习，理论上比 AvgPool 更好地保留小目标信息 |

#### 方案 1C：1x1 投影 + 标准 Conv 下采样（较重，不推荐端侧）

```
fuse_stage0: Conv2d(48, 256, 1x1) + Conv2d(256, 256, 3x3, stride=4, padding=1)
             参数: 12,288 + 589,824 = 602,112

fuse_stage1: Conv2d(96, 256, 1x1) + Conv2d(256, 256, 3x3, stride=2, padding=1)
             参数: 24,576 + 589,824 = 614,400
```

| 指标 | 值 |
|---|---|
| 新增参数 | **1,216,512 (~1.2M)** |
| 占模型总参数比 | 12.7% |
| 端侧友好度 | 偏重，不推荐 |

#### 方案一总结

| 子方案 | 新增参数 | 占比 | 端侧友好 | 效果预期 |
|---|---|---|---|---|
| **1A: 1x1 + AvgPool** | **37K** | **0.38%** | 非常友好 | 好，先试 |
| 1B: 1x1 + DWConv | 44K | 0.46% | 友好 | 稍好于 1A |
| 1C: 1x1 + Conv | 1.2M | 12.7% | 不推荐 | 最好但太重 |

### 代码改动位置

`edge_sam/modeling/rep_vit.py` 中 `RepViT.__init__` 的 fuse 分支和 `forward` 方法。

## 方案二：升级编码器 m1 → m2

### 核心思路

直接使用更宽通道的编码器变体。

| | m1 | m2 | m3 |
|---|---|---|---|
| Stage 0 通道 | 48 | 64 | 64 |
| Stage 1 通道 | 96 | 128 | 128 |
| Stage 2 通道 | 192 | 256 | 256 |
| Stage 3 通道 | 384 | 512 | 512 |
| Block 总数 | 26 | 23 | 34 |
| 估计参数量 | ~9.6M | ~15-18M | ~25M+ |

### 注意事项

- 权重不兼容，需要**重新蒸馏**或从 RepViT m2 预训练权重开始微调
- 参数量增加约 60-90%，端侧推理速度会受影响
- 修改点：`build_sam.py` 中 `arch="m1"` → `arch="m2"`

### 建议

如果方案一效果不够再考虑。对于端侧部署需要评估 m2 的推理耗时是否可接受。

## 方案三：增强 Mask Decoder

### 当前 Decoder 配置

```python
TwoWayTransformer(
    depth=2,              # 2 层 transformer
    embedding_dim=256,
    mlp_dim=2048,
    num_heads=8,
)
# 上采样: 2 层 ConvTranspose2d，4x 上采样 (64x64 → 256x256)
```

### 可调整项

| 修改 | 当前值 | 建议值 | 新增参数估计 |
|---|---|---|---|
| Transformer depth | 2 | 3~4 | ~2.5M/层 |
| MLP dim | 2048 | 4096 | ~2M |
| 上采样层数 | 2 (4x) | 3 (8x, 输出 512x512) | ~100K |

### 注意事项

- Decoder 增加 depth 参数量增长较快，需要权衡端侧推理耗时
- 增加上采样层对小目标边界精度有帮助，参数增量较小
- 与方案一可组合使用

---

## 方案四：LoRA 微调策略（不改架构，防遗忘）

### 核心思路

项目已支持 LoRA（`transformer.py` 中集成了 `loralib`）。小目标掉 20 个点很可能包含**灾难性遗忘**因素。

### 策略

1. 冻结编码器 (`freeze=True`)，保留原始预训练的小目标特征提取能力
2. 只用 LoRA 微调 decoder 的 attention 层（当前 rank=16）
3. 训练参数量极小，不影响推理（LoRA 可合并到原始权重中）

### 优势

- **零额外推理参数**（LoRA 权重合并后）
- 保留预训练模型在小目标上的能力
- 可与方案一组合使用

---

## 推荐实施路线

```
第一步: 方案四 (LoRA) + 方案 1A (1x1 + AvgPool)
        → 新增参数 ~37K，解决遗忘 + 补充高分辨率信息
        │
        ├─ 效果满意 → 完成
        │
        └─ 效果不够
              │
              ▼
第二步: 升级为方案 1B (1x1 + DWConv)
        → 新增参数 ~44K，可学习的下采样
        │
        ├─ 效果满意 → 完成
        │
        └─ 效果不够
              │
              ▼
第三步: 考虑方案二 (m1 → m2) 或方案三 (增强 Decoder)
        → 需要重新评估端侧推理耗时
```
